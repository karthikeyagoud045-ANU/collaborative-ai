# Architecture Overview

The **Ultimate Vibe Coder** is a robust, highly-concurrent, collaborative coding platform. This document outlines the high-level architecture, primary components, and data flow of the application.

## High-Level Architecture

The system is composed of four primary layers:
1. **Frontend Layer (Next.js & React 19):** Handles the UI, state management, and real-time multiplayer rendering.
2. **Execution Layer (WebContainers):** A micro-operating system running entirely inside the browser's WebAssembly sandbox.
3. **Collaboration & Sync Layer (Yjs & WebSocket):** A custom Node.js WebSocket server routing Conflict-Free Replicated Data Types (CRDTs).
4. **Persistence Layer (Supabase):** PostgreSQL database storing Yjs document states and user settings.

---

## Key Components

### 1. WebContainers (Execution Layer)
WebContainers allow the IDE to run a full Node.js environment directly in the browser.
- **Process Management:** The frontend tracks spawned processes (e.g., `npm run dev`) and intercepts terminal output.
- **File Synchronization:** Changes made in the Monaco Editor are aggressively debounced and synced into the WebContainer virtual file system to trigger hot-reloading (HMR).
- **Security:** WebContainers run within strict Cross-Origin Isolation (COOP/COEP) headers, ensuring sandbox security.

### 2. Yjs & CRDTs (Collaboration Layer)
Yjs is the backbone of multiplayer collaboration.
- **`monaco` (Y.Text):** Synchronizes the active file's text content.
- **`chat` (Y.Array):** Synchronizes the team chat messages.
- **`aiQueue` (Y.Map):** Tracks pending agentic actions across users.
- **Awareness Protocol:** Tracks user presence, cursor locations, and active selections, painting them onto the Monaco Editor instance via `y-monaco`.

### 3. Agentic Loop & HITL (AI Layer)
The platform features an autonomous AI agent capable of multi-step execution.
- **Agent Loop (`src/lib/agent-loop.ts`):** Listens to the `aiQueue`. When a task is picked up, it streams responses from an LLM (OpenAI, Anthropic, or Gemini).
- **Tool Use:** The agent can issue commands (e.g., `write_file`, `run_command`).
- **Human-in-the-Loop (HITL):** Destructive actions generated by the AI (like overwriting a file or running a shell command) are blocked by default. They are pushed to the `pendingAgentActions` queue and broadcasted to all users in the room. Any user can approve or reject the action, creating a transparent, democratic AI collaboration experience.
- **Conflict Resolution:** If multiple AI agents try to edit the same file simultaneously, the system traps the changes and presents a Git-like diff view (`ConflictResolver.tsx`) to the users.

### 4. WebSocket Server (`server/index.js`)
A lightweight, fast Node.js server.
- **Routing:** Connects clients together based on the URL path (the `roomId`).
- **Persistence:** In production, it debounces document updates and pushes base64-encoded Yjs state vectors to Supabase. Upon room creation, it hydrates the Yjs document from Supabase.
- **Health Checks:** Features an integrated HTTP `/health` endpoint for Docker/Fly.io readiness probes.

```text
│   ├── Dockerfile            # Docker configuration for WS server
│   └── fly.toml              # Fly.io deployment config
├── src/
│   ├── app/                  # Next.js App Router (Pages & API routes)
│   │   ├── room/[roomId]/    # Core IDE interface
│   │   └── api/              # AI endpoints, Vault, Github integrations
│   ├── components/           # React Components (Editor, Chat, Panels, Modals)
│   ├── hooks/                # Custom React Hooks (useToast, useAgentApprovals)
│   └── lib/                  # Core Logic (WebContainer, Yjs, AI Loop, Logger)
├── supabase/                 # Supabase schema and migrations
└── next.config.ts            # Next.js Configuration (COOP/COEP headers)
```

## Data Flow (File Edit Example)
1. **User Edit:** User types in the Monaco Editor.
2. **Yjs Sync:** The keystroke updates the local `Y.Text` instance. Yjs encodes this as a binary update and sends it over the WebSocket.
3. **Broadcast:** The WebSocket server receives the binary update and broadcasts it to all other users in the room.
4. **Remote Update:** Other clients receive the update. Yjs merges it, and Monaco reflects the new text.
5. **WebContainer Sync:** The local browser debounces the text change and writes the file into the WebContainer virtual file system.
6. **HMR:** The running `npm run dev` process in the WebContainer detects the file change, rebuilds, and pushes an HMR update to the `LivePreview` iframe.
